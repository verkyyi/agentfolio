# Generic Evolve: Make the Scaffold Work for Any Repo

## Problem

Evolve and other workflows are hardcoded for agentfolio. Research sources watch Claude Code harness repos. Analysis steps evaluate an Astro profile page, track agentfolio growth metrics, and monitor agentfolio adopters. Adopters who add the scaffold to their own repo get irrelevant research, wasted tokens, and broken steps.

## Solution

Two-phase system: interactive onboarding generates a config file that gates and customizes all autonomous workflows.

## Phase 1: Interactive Onboarding (CLI Session)

Triggered by the user running Claude Code in their repo during setup. NOT triggered by cron or workflow dispatch — requires a human in the loop for confirmation.

### Steps

1. Copy workflows, state files, skills from agentfolio
2. Analyze repo: detect stack (language, frameworks, dependencies), purpose, maturity
3. Estimate workflow fit for each of the 11 workflows:
   - **HIGH** — works as-is, will provide value
   - **MEDIUM** — works but may need tuning
   - **LOW/SKIP** — not applicable to this repo
4. Present estimates to the user. Ask about gaps directly:
   - "How do you deploy?" (if no deploy config detected)
   - "What test command should reviewer run?" (if no test framework detected)
   - "Is this a public or private project?" (determines growth.yml relevance)
5. Auto-discover research sources based on detected stack:
   - Identify key dependencies from package.json / requirements.txt / go.mod
   - Map to GitHub repos for those frameworks/libraries
   - Always include `verkyyi/agentfolio` for scaffold updates
6. Write confirmed config to `state/evolve_config.md`
7. Disable SKIP workflows (comment out cron triggers or remove files)
8. Commit and push

### Config File Format

```markdown
# Evolve Config
# Generated by onboarding CLI session. Human edits welcome.
# Delete this file to re-trigger onboarding.

## Repo Profile
Stack: Python, FastAPI, PostgreSQL
Purpose: SaaS API backend
Maturity: Early (12 commits, 3 contributors)

## Workflow Fit
evolve: HIGH
triage: HIGH
coder: HIGH
reviewer: HIGH
watcher: HIGH
deploy: SKIP — user deploys via Docker to AWS ECS
growth: SKIP — private repo
feedback-learner: MEDIUM

## Research Sources
- fastapi/fastapi
- encode/databases
- verkyyi/agentfolio

## User Answers
Deploy: Docker → AWS ECS (not GitHub Pages)
Tests: pytest, runs via `make test`
Public: No — private repo
```

## Phase 2: Autonomous Evolution (Cron)

### Gate Mechanism

Every workflow checks for `state/evolve_config.md` before running. If missing, exit immediately with zero tokens spent:

```yaml
- name: Check onboarding complete
  run: |
    if [ ! -f state/evolve_config.md ]; then
      echo "Onboarding not complete. Run: claude 'Set up agentfolio in this repo. Follow https://github.com/verkyyi/agentfolio/blob/main/docs/onboarding.md'"
      exit 0
    fi
```

Added to every workflow, before the Claude Code step.

### Evolve Behavior (with config)

Evolve reads `state/evolve_config.md` and:

- **Research**: Fetches commits from auto-discovered sources + `verkyyi/agentfolio`. No hardcoded gstack/ECC/etc unless relevant to the repo.
- **Pipeline health** (Step 2a): Runs as-is — generic, checks workflow failures.
- **Design evaluation** (Step 2b): Skipped unless config says the repo has a frontend.
- **Growth metrics** (Step 2c): Skipped unless config says public repo.
- **Adoption tracking** (Step 2d): Skipped unless config says template/scaffold repo.
- **SEO** (Step 2e): Skipped unless config says public site.
- **Human intent analysis** (Step 2f): Runs as-is — generic.
- **Periodic re-check**: Every 7 days (or when file structure changes significantly), evolve compares current repo structure against the config. If a mismatch is detected (new framework added, deploy config appeared), it creates an issue: "Repo structure may have changed — run a CLI session to update evolve config."

### Other Workflows

- **triage, coder, reviewer, watcher**: Already generic — issue/PR-driven, no config changes needed. Gate check added but behavior unchanged.
- **deploy**: Skipped if config says SKIP. Otherwise runs as-is.
- **growth**: Skipped if config says SKIP.
- **feedback-learner**: Runs if MEDIUM or HIGH.
- **discover, claude-task, analyze**: Generic, run as-is.

## Scope

### Changes Required

1. **New file**: `state/evolve_config.md` — generated by onboarding, read by workflows
2. **All 11 workflows**: Add gate check step (3 lines each)
3. **evolve.yml prompt**: Replace hardcoded sources/steps with config-driven logic. The prompt reads `state/evolve_config.md` and adapts its behavior.
4. **docs/onboarding.md**: Add repo analysis, fit estimation, and interactive Q&A steps
5. **Onboarding prompt**: The Claude CLI session follows the updated onboarding doc

### Not Changed

- Workflow YAML structure (triggers, permissions, concurrency) — unchanged
- State file format (agent_log, research_log, etc.) — unchanged
- CLAUDE.md autonomy rules — unchanged
- Skills — unchanged

## Risk

- **Auto-discovery accuracy**: Claude may misjudge the stack or pick irrelevant sources. Mitigated by human confirmation during onboarding.
- **Config staleness**: Repo changes but config doesn't. Mitigated by periodic re-check + issue creation.
- **Gate failure**: If `state/evolve_config.md` is accidentally deleted, all workflows stop. This is by design — it forces re-onboarding rather than running blind.
